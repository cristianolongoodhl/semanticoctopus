INTRODUCTION

The Semantic Octopus is a java application which allows one to publish a SPARQL endpoint providing a knowledge base obtained by aggregating multiple RDF knowledge bases from the web and performing subsequent reasoning (OWL-DL). The Semantic Octopus is based on jena and pellet (ansell version). We use it in conjunction with a php proxy for security reasons. A running instance of Semantic Octopus is currently used to provide the opendatahacklab knowledge base and to generate the contents of some pages (see for example the opendatahacklab projects page.)

COMPILE AND INSTALL

Semantic Octopus can be compiled from sources with maven (just run mvn install in the sources root). After compilation you will find a semanticoctopus-[vers]-jar-with-dependencies.jar file in the maven-generated target directory. The endpoint can be setup with the following command: 

java -jar semanticoctopus-version-jar-with-dependencies.jar endpointhost enpointport kburl1 [kburl2 [... kburln]]

where

  1. version is the semantic octopus version;
  2. endpointhost is the host name under which the aggregated ontology will be provided;
  3. enpointport is the endpoint port;
  4. kburl1, ..., kburln are the URLs of the ontologies to be aggregated.

For example, the following will start a sparql endpoint at port 9001 of localhost which provide the main opendatahacklab projects and events knowledge base. 

java -jar semanticoctopus-0.2.0-jar-with-dependencies.jar http://localhost 9000 http://opendatahacklab.org/site/odhl.owl

Once started, the semantic octopus interactive console is displayed. With typing help you can get all the available commands, which are explained in details in the following section. To start using the endpoint just enter build in the console and wait until aggregation and reasoning ends.

USAGE AND LIFECYCLE

The following commanda are available in the semantic octopus console:

  • help prompt all available commands;
  • list print the list kburl1, ..., kburln of ontology URLs which will partecipate in forming the provided knowledge base;
  • build start aggregation and reasoning;
  • state print the current state of the semantic octopus instance;
  • exit finish the execution of the instance and free all resources.

A semantic octopus instance can be in one of the following states:

  • IDLE is the initial state, non aggregation has take place and the knowledge base provided via the endpoint is the empty one;
  • BUILDING state means that the octopus has started downloading the ontologies, performing reasoning and checking aggregated ontology consistency;
  • READY state means that aggregation was succesful and the resulting aggregated ontology is consistent;
  • ERROR state is entered when the building phase failed because of some error downloading ontologies of because the resulting ontology was not consistent.
  • DISPOSED is the final state, reached when the user enter the exit command and all resources are free.

Note that any command is just ignored in BUILDING state. In addition, in BUILDING and ERROR state the ontology previously generated by the last succesful build is server, until a new consistent aggregated ontology is available due to a succesful build.
